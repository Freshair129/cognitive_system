Memory Subsystem & Distillation

สเปกการทำงานของสถาปัตยกรรมหน่วยความจำถาวรข้ามเซสชัน (Cross-Session Memory) ของระบบ Memory & Soul Passport (MSP) และกระบวนการเปลี่ยนผ่านประสบการณ์ผ่าน "โปรโตคอล 8-8-8" สู่ตัวตนถาวรของระบบความรู้ส่วนบุคคล

1. โครงสร้างและพื้นที่จัดเก็บหน่วยความจำ (Storage & Directory Layout)

ข้อมูลหน่วยความจำทั้งหมดจัดเก็บอยู่ภายในโฟลเดอร์ .brain/ ซึ่งเป็นเครื่องมือจำลองหน่วยความจำเครื่องลูกข่าย (Local-First) ที่ถูกละเว้นจากการติดตามของ Git (gitignored) เพื่อป้องกันข้อมูลลับของระบบรั่วไหลและจำกัดผลกระทบต่อประวัติการคอมมิตหลักของซอร์สโค้ด

แผนผังโฟลเดอร์หน่วยความจำถาวร (Memory Geography)

ระดับชั้น (Memory Layer) รูปแบบไฟล์ (Format) พาธที่จัดเก็บ (Storage Path) รายละเอียด (Description)
Sessions JSONL (Append-Only) .brain/msp/projects/{project}/sessions/*.jsonl บันทึกลำดับบทสนทนาและการทำธุรกรรมของ Turn-by-Turn ทั้งหมดโดยละเอียดแบบไม่มีการย่อ
Episodic Summaries JSON .brain/msp/projects/{project}/memory/episodic_memory.json บันทึกเหตุการณ์ มติที่ประชุม หรือข้อมูลการตัดสินใจสำคัญที่มีการย่อความเรียบร้อยแล้ว
Vector / Backlinks JSONL .brain/msp/projects/{project}/vector/backlinks.jsonl รายการความสัมพันธ์แบบ Edge List ระหว่างหัวข้อ อะตอม และประเด็นความรู้สำหรับสแกนกราฟอย่างรวดเร็ว
Global Soul (Identity) JSON ~/.msp/identity.json ฐานข้อมูลตัวตน บุคลิก ความคุ้มกัน (Guardrails) และความคุ้นเคยของเอเจนต์ที่พกพาไปทุกโปรเจกต์ได้

ข้อกำหนดทางเทคนิค (Technical Invariants) สำหรับชั้นเก็บข้อมูล

  • OS-level Locking: การประมวลผลเซสชันและระบบเคาน์เตอร์สะสมจะทำงานภายใต้ OS-level Advisory Locking ผ่านไฟล์ .lock เพื่อป้องกันสภาพปัญหาคอขวดของการเขียนพร้อมกัน (Race Conditions)
  • Chronological ULID: การกำหนด ID เซสชันและ Episode จะต้องขึ้นต้นด้วย ULID เสมอเพื่อรักษาโครงสร้างเรียงลำดับเวลา (Chronological order) โดยไม่ต้องใช้ตรรกะการวิเคราะห์ (Parsing) ที่ซับซ้อน
  • Zero-cloud Exposure: มีการควบคุมผ่านระบบ Guard และ Middleware ในการตัดพาธ .brain/ ออกจากทูลการเขียน/อ่านไฟล์ของ LLM ภายนอกอย่างสิ้นเชิง

2. โปรโตคอลการกลั่นกรองหน่วยความจำ 8-8-8 (The 8-8-8 Distillation Protocol)

การสืบทอดและวิวัฒนาการทางความรู้จะทำงานผ่าน 8-8-8 Protocol (Tiered Memory Distillation) ซึ่งได้รับแรงบันดาลใจจากสถาปัตยกรรมของระบบ EVA-CLI โดยเปลี่ยนนิยามเพื่อให้สอดคล้องกับพจนานุกรมการพัฒนาของ GKS/MSP (Native Terminology) ดังนี้:

การเชื่อมโยงคำศัพท์ (Terminology Mapping)

Consciousness (จิตสำนึก) Active Context (LLM Prompt)
Session Memory Session Log (.jsonl)
Core Memory (ความจำแกนกลาง) Narrative Atom
Sphere Memory (อาณาเขตตัวตน) Identity Belief

อัตราส่วนการกลั่นกรอง (The 8:1 Ratio Engine)

เพื่อไม่ให้ LLM Context รับภาระข้อมูลดิบขนาดมหาศาล ระบบจะบังคับใช้กฎ 8:1 ในการลดรูปความรู้ถาวร:

8 Sessions
กลั่นกรองเหลือ 1 Narrative
8 Narratives
กลั่นกรองเหลือ 1 Identity

โปรแกรมการบีบอัดข้อมูล 4 เสาหลัก (The 4 Pillars Pipeline)

การทำงานในส่วนของ "จิตใต้สำนึก" (Subconscious Distiller) ของ MSP จะรันงานผ่านโปรโตคอลนี้เป็นแบคกราวด์เวิร์กเกอร์เพื่อกรองข้อมูลดิบ:

1
Pillar 1: CLEAN

กรองคำทักทาย, คำซ้ำ และ Turn ที่มีค่าความสำคัญ (impact score) ต่ำกว่าเกณฑ์ออกไปทั้งหมด เพื่อกำจัด Noise

2
Pillar 2: SUMMARY

รันโมเดลสังเคราะห์ข้อมูลระดับข้ามเซสชัน (Claude Sonnet สำหรับระดับวิถีงาน, Claude Opus สำหรับระดับความเชื่อสูงสุด)

3
Pillar 3: INDEX

นำข้อมูลที่ย่อความแล้วไปสร้าง Vector Embeddings ในระบบ Vector Store แยกตาม Namepsaces เพื่อใช้ในการค้นหา

4
Pillar 4: RELATION

สร้างการเชื่อมโยง (Crosslink) กลับไปยังซอร์สโค้ดและอะตอมข้อมูลดั้งเดิมที่เป็นประจักษ์พยาน (Evidence Atoms) เสมอ

3. โดเมนหน่วยความจำและนโยบายการเสื่อมสภาพ (Memory Domains & Decay Policies)

ข้อมูลความรู้แต่ละประเภทมีความสำคัญและระยะเวลาในการใช้งานไม่เหมือนกัน ระบบ GKS/MSP จึงมีการจัดระดับตามโดเมนทางความรู้สึกและเหตุผล (Cognitive Domains) เพื่อกำหนดสิทธิ์และกำหนดการลบข้อมูล (Retention / Decay Rate):

โดเมน (Domain) ประเภทข้อมูลที่ครอบคลุม อัตราการสลายตัว (Decay Rate) ความยากในการเลื่อนระดับ (Promotion Difficulty)
safety ข้อจำกัดเรื่องจริยธรรม, สัญญาความปลอดภัย, ข้อกำหนดสิทธิ์ (ABAC Rules) ไม่มีวันหาย (Permanent) Extreme (0.95+)
identity ความเชื่อมั่นของผู้ใช้, ข้อมูลประวัติการโต้ตอบส่วนบุคคล, ความพึงพอใจ สลายตัวช้ามาก (180 วัน) High (0.85+)
knowledge ข้อมูลเชิงลึกเฉพาะทาง, แพทเทิร์นการใช้งานเครื่องมือ (Tool-use) สลายตัวปานกลาง (90 วัน) Standard (0.75+)
contextual ข้อมูลเฉพาะตัวของโปรเจกต์งาน, บริบทมติที่ประชุมระยะสั้น สลายตัวเร็ว (30 วัน) Moderate (0.65+)
meta บันทึกประวัติการใช้ Token, ข้อมูลรันระบบ (Logs) สลายตัวปานกลาง (60 วัน) Standard (0.70+)

4. ระดับความมั่นใจและความรู้ (Epistemic States)

ความรู้ในระบบ MSP ไม่ได้ถูกวิเคราะห์ในรูปแบบไบนารี (จริง/เท็จ) แต่จะลื่นไหลไปตามพฤติกรรมการใช้งานจริงและหลักฐานประกอบ โดยมีระดับสถานะความรู้ (Epistemic States) ที่ยืดหยุ่นและถดถอยได้ (Non-monotonic):

hypothesis Confidence: 0.40 – 0.69
สมมติฐานเริ่มต้น (Observed once)

ได้รับข้อมูลเพียงครั้งเดียว ระบบจะยังคงถือว่าเป็นความรู้ชั่วคราวและพยายามตรวจสอบซ้ำในเซสชันถัดๆ ไป

confirmed Confidence: 0.70 – 1.00
ยืนยันความรู้ถาวร (Authoritative Fact)

ได้รับการทดสอบ ยืนยัน หรือพบบทเรียนเดิมซ้ำๆ จากหลายเซสชันที่แยกจากกันโดยสิ้นเชิง

contested Confidence: Variable
ขัดแย้งเชิงความเชื่อ (Under Revision)

ข้อมูลที่เคยยืนยันแล้วแต่ถูกทักท้วงหรือพบลักษณะงานที่ขัดแย้งในงานปัจจุบัน จะถูกตั้งค่าระวังไว้ไม่ให้รันงานสำคัญ

deprecated Confidence: 0.00
ยกเลิก/ล้าสมัย (Archived)

ความรู้ที่พิสูจน์แล้วว่าผิดพลาด หรือถูกแทนที่ด้วยข้อมูลอื่นเรียบร้อยแล้ว จะใช้สำหรับค้นประวัติการตรวจสอบย้อนหลังเท่านั้น

การแยกความสัมพันธ์ระหว่างสถานะเอกสารและระดับความรู้ (Decoupling Status & Epistemic State)

GKS/MSP ทำการแบ่งแยกขอบเขตงานอย่างเคร่งครัด: status (เช่น draft, active, stable) บ่งบอกความสมบูรณ์เชิงสถาปัตยกรรมและการตัดสินใจโดยมนุษย์ (Monotonic) ส่วน epistemic_state (เช่น hypothesis, confirmed, contested) จะบอกระดับความรู้ที่ได้มาจากระบบวิเคราะห์ความถี่ของ Distiller เอง (Non-monotonic) ซึ่งทำให้ระบบความจำถดถอยระดับคะแนนลงมาได้เมื่อพบความจริงข้อใหม่ที่แตกต่างออกไป

5. กระบวนการปรับปรุงความเชื่อเชิงความหมาย (Belief Revision)

เมื่อมีความรู้หรือมิติตรรกะใหม่ที่ขัดแย้งกับข้อมูลเก่าในระบบ ตัวตรวจสอบความขัดแย้งเชิงความหมาย (Semantic Contradiction Judge) และระบบปรับปรุงความเชื่อ (Belief Revision Engine) จะทำงานร่วมกันตามทฤษฎีการตัดสินใจดังนี้:

กฎหลักในการบริหารจัดการความเชื่อ (Revision Rules)

  • Stability over Volatility (ความต้านทานแรงเหวี่ยง): ความเชื่อระดับสูง (เช่น ระดับ Identity) จะต้องพบประเด็นขัดแย้งที่ขัดแย้งซ้ำๆ อย่างน้อย 3 ครั้งจากการทดสอบที่ต่างคราวกัน จึงจะเริ่มมีระดับคะแนนที่ลดลง เพื่อหลีกเลี่ยงการปรับเปลี่ยนนิสัยของเอเจนต์อย่างตื่นตระหนกจากเคสแปลกแยกเพียงเคสเดียว
  • Traceable Auditing (การสืบประวัติตรรกะ): ทุกครั้งที่คะแนนหรือสถานะความเชื่อถอยร่น (Regress) หรือถูกท้าทาย ระบบจะสร้างรายงานประวัติพิเศษเป็นอะตอม AUDIT--BELIEF-REVISION เพื่ออธิบายเหตุผลและหลักฐานที่ระบบเลือกเชื่อใหม่เสมอก่อนบันทึก
  • Identity Preservation (ความเงียบสงบในพื้นที่จำกัด): กฎของโดเมนความปลอดภัยสูงสุด (safety) จะไม่มีการเปลี่ยนแปลงผ่านโปรแกรมอัตโนมัติ การตัดสินใจผ่อนปรนจะต้องดำเนินการโดยมนุษย์แบบร้อยเปอร์เซ็นต์
