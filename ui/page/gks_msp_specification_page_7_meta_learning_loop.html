Meta Learning Loop (MLL)

ข้อกำหนดเชิงเทคนิคสำหรับ "วงจรการเรียนรู้อนุมานกลับ" (Self-Refining Cognitive Loop) ของระบบ GKS และ MSP ทำหน้าที่เปรียบเสมือนสะพานเชื่อมระหว่างการปฏิบัติงานจริงในสายการผลิต (P5 CODE) เพื่อตกตะกอนหลักการกลับเข้าสู่ฐานความรู้ถาวรของเครื่อง

1. วงจรการสังเคราะห์ความรู้สากล (The Synergy Cycle)

Meta Learning Loop (MLL) ถือเป็นชั้นบริหารจัดการความรู้ (Knowledge Governance Engine) ขั้นสูงสุดที่ทำหน้าที่ปิดวงลูปการทำงานในระบบ GKS/MSP หากปราศจาก MLL ระบบจะทำได้เพียงประมวลผลงานตามสั่งเป็นครั้งคราว (One-shot Execution) โดยปราศจากการสะสมภูมิปัญญา

MLL คลี่คลายปัญหา amnesia (ความทรงจำเสื่อมถอย) ข้ามโปรเจกต์ด้วยการขับเคลื่อนวงจรที่เรียกว่า The Synergy Cycle ซึ่งประกอบด้วยกระบวนการไหลเวียนข้อมูลสองส่วนประสานงานกัน:

ผังปฏิสัมพันธ์วงจรร่วมพลังของ MLL (The Synergy Cycle)

FORWARD PATH (7P)
Block Assembly (สร้างจากสเปก)

การสร้างและวางโครงสร้างอย่างเป็นระเบียบจากบนลงล่างตามนโยบาย Doc-before-code โดยมีมนุษย์กุมทิศทาง:

Intent → CONCEPT → ADR → BLUEPRINT → Microtask → CODE

REVERSE PATH (12S)
Block Decomposition (อนุมานจากโค้ด)

กระบวนการดึงข้อเท็จจริงตามหลักสากลจากรหัสที่แก้ไขและพฤติกรรมทำงานจริงผ่านเครื่องมือวิเคราะห์กราฟ:

Source Code → AST parsing → Symbol Graph → Execution Trace

2. กลไกสร้างทักษะอัตโนมัติ (Skill Creator: Hermes Engine)

กลไก Skill Creator (หรือรหัสเครื่องยนต์ Hermes) ทำหน้าที่เรียนรู้พฤติกรรมความสำเร็จในอดีตมาสร้างเป็นชุดคู่มือปฏิบัติงานอัตโนมัติ (SKILL--) เพื่อลดภาระและประหยัดงบประมาณโทเค็นของเอเจนต์ในลูปถัดไป โดยรันงานเป็น Background Worker ภายใต้เงื่อนไขอย่างเคร่งครัด (FEAT--SKILL-CREATOR):

1
Task Success Gate

ไมโครทาสก์ระดับปฏิบัติการ (T1) ได้รับการประเมินยืนยันผลสอบผ่านชุดรันเทส (Acceptance Test Passed) ในครั้งแรกหรือมีการรีไทร์น้อยกว่าเกณฑ์ที่กำหนด

2
Workflow Complexity

ลำดับการทำงานจริง (Execution Trace) มีความซับซ้อนสูงกว่าเกณฑ์วิกฤต เช่น มีการเรียกใช้โมเดลภาษาติดต่อกันมากกว่า 3 ครั้งย่อย หรือเรียกใช้ API ซับซ้อน

3
Novelty Verification

ลูปสแกนความสัมพันธ์ (Pattern matching) ไม่พบพฤติกรรมนี้หรือหัวข้อใกล้เคียงในดัชนีคลังความรู้ทักษะปัจจุบัน (gks/skills/)

ข้อกำหนดคลังพักรอตรวจสอบความเชื่อมั่น (Human-in-the-loop Staging Area)

เพื่อรักษาความน่าเชื่อถือสูงสุดของฐานความรู้สัญชาตญาณ (SSOT) ห้ามไม่ให้ระบบ MLL บันทึกทักษะลงพื้นที่ถาวรโดยอัตโนมัติเด็ดขาด (Never Auto-write to Canon) ผลผลิตจากโปรเจกเตอร์ Hermes จะถูกจัดวางอย่างปลอดภัยในโฟลเดอร์สมัครเล่น .brain/msp/projects/{project}/candidates/SKILL--*.md เสมอ เพื่อรอให้มนุษย์เข้ามาตรวจสอบและคอมมิตผ่านระบบ Pull Request ปิดท้ายตามกฎหมายเหล็กของสถาปัตยกรรม

## Behavioral Steps

1. เมื่อประมวลผล Express endpoint ค้นหาการใช้ไวยากรณ์ `req.body`
2. บังคับย้ายตัวแปรผ่าน middleware ตรวจสอบ UCF Attribute Bag เสมอ
3. รันเช็ค Pattern ลิสต์ด้วย PII Redaction ก่อนส่งคำขอยื่น LLM ภายนอก

3. กฎความสมบูรณ์เชิงสัจพจน์ 4 มิติ (The 4D Completeness Matrix)

ความรู้ระดับรากแก้ว (Master-tier Genesis Block) ที่พร้อมจะได้รับการโหลดเข้าไปเป็น System Prompt เสมือนสัญชาตญาณดิบของระบบ จะต้องผ่านเกณฑ์ประเมินด้านความสมบูรณ์แบบรอบด้านสี่มิติ 4D Completeness Check เพื่อป้องกันไม่ให้เกิดความรู้ลอยๆ ที่เอาไปปฏิบัติจริงไม่ได้ หรือกฎปฏิบัติการที่ไร้ความหมายเชิงบริบท:

มิติแห่งความรู้ (Dimension) คำขึ้นต้นอะตอม (Atom Prefix) วัตถุประสงค์ (Functional Purpose) สัญชาตญาณที่ตอบโจทย์
1. Algo (ตรรกะกรรมวิธี) ALGO-- ตรรกะการประมวลผลเชิงลำดับ, ขั้นตอนทางคณิตศาสตร์, หรือโครงสร้างซูโดโค้ด "เรารันงานทีละขั้นตอนด้วยอัลกอริทึมใด?"
2. Concept (ความรู้สึกเชิงลึก) CONCEPT-- นิยามเชิงนิรุกติศาสตร์, คำอธิบายเป้าหมายทางสถาปัตยกรรม, วิสัยทัศน์ระบบ "นิยามและโครงสร้างแนวคิดนี้มีความหมายว่าอะไร?"
3. Frame (สัญญากรอบความคิด) FRAMEWORK-- ระเบียบวินัยภาพกว้างของโมเดลความคิด, การพึ่งพาและความปลอดภัยระดับระบบ "ขอบเขตและระเบียบภาพรวมนี้อยู่ภายใต้ข้อกำหนดใด?"
4. Proto (แนวทางรันไทม์) PROTOCOL-- / PROTO-- การจัดวางอินเตอร์เฟซ API, ข้อกำหนดการโต้ตอบข้าม FFI, กฎสายพานคิวงาน "การจัดเรียงและการประมวลผลรันไทม์ต้องใช้หน้าตาแบบไหน?"

กฎเหล็กการเลื่อนระดับผ่านด่านตรวจ (Completeness Validation Gate)

4. การวิเคราะห์ความเสถียรด้วยฉันทามติหลายค่าย (Multi-Model Stability Check)

เพื่อลบล้างปัญหาเอเจนต์เขียนเอกสารและรหัสมักง่ายแบบพึ่งพาสำนวนภาษาที่เฉพาะเจาะจงของบางค่าย (Model-bias) MLL บังคับใช้โปรแกรมตรวจสอบความมั่นคงของสัญกรณ์แบบ **Multi-Model Consensus** ผ่านกลไกประเมินระดับความนิ่งเชิงลึก:

STEP 1
Input Definition

ระบบรับอะตอมดราฟต์ที่เจเนอเรตขึ้นมาจาก candidates โฟลเดอร์มาสกัดเอาเฉพาะส่วนบอดี้ข้อมูล Markdown

STEP 2
Parallel Execution

ยิงคำสั่งวิเคราะห์ไปหา 3 โมเดลค่ายต่างกัน: Gemini 1.5 Pro, Claude 3.5 Sonnet และ Local Ollama Qwen

STEP 3
Core Summary

สั่งให้ทุกโมเดลตอบสรุปใจความสำคัญของเอกสารนั้นกลับมาในรูปแบบสั้นๆ บรรจุข้อเท็จจริงประจักษ์พยาน

STEP 4
Cosine Similarity

นำเอาสรุปของทั้ง 3 ค่ายมาแปลงเวกเตอร์ด้วยตัวแรป Nomic Embedder และวัดระยะห่างความเหลื่อมล้ำ

เงื่อนไขค่าความพึงพอใจฉันทามติ (Consensus Cutoff Threshold)

ค่าความนิ่งเชิงลึก (stability_score) จะคำนวณผ่านการหาค่าเฉลี่ยมุมเวกเตอร์ (Average Cosine Similarity) ของทั้ง 3 โมเดลย่อย:

similarity = (cos_sim(A, B) + cos_sim(B, C) + cos_sim(A, C)) / 3
Score ≥ 0.85: STABLE (ผ่าน)
มีความเสถียรเชิงความหมายข้ามค่ายโมเดลสูงมาก อนุญาตให้ขึ้นคิวโปรโมตเป็นความรู้ระดับ Master ได้
Score < 0.85: REJECT (พัง)
มีความกำกวมเชิงภาษาหรือการตีความ แย้งกันชัดเจน บังคับดีดกลับไปปรับปรุงสัญกรณ์ใหม่

5. ตัวควบคุมความขัดแย้งเชิงความหมาย (Semantic Drift & Tension Event Logging)

เมื่อซอร์สโค้ดในรันไทม์จริง (สกัดผ่าน 12-Stage DAG) เริ่มมีแนวโน้มการขัดแย้งกับหลักสัจพจน์และนโยบายที่บันทึกไว้ใน GKS Atoms (เช่น ในสเปกบอกว่าใช้สิทธิ์ระดับ T1 แต่โค้ดจริงมีคำสั่งเรียกใช้โมเดล Tier 3) ระบบจะเกิดปรากฏการณ์ที่เรียกว่า **Semantic Drift (ความรู้เบี่ยงเบน)**
