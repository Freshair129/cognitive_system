Universal Context Framework & Policy Engine

ข้อกำหนดระบบการกรองบริบทข้อมูลและการบังคับใช้นโยบายความปลอดภัยเชิงรุก (Zero-Trust Architectural Context) ของระบบ Memory & Soul Passport (MSP) เพื่อแยกแยะสิทธิ์ ควบคุมขอบเขต และจำกัดการเข้าถึงข้อมูลอ่อนไหวระดับบุคคล

1. สถาปัตยกรรม UCF (Universal Context Framework Architecture)

Universal Context Framework (UCF) เป็นเฟรมเวิร์กควบคุมการเข้าถึงข้อมูลและการไหลของข้อมูล (Information Flow Control) ที่อยู่ในชั้นเหนือหน่วยจัดเก็บข้อมูล GKS โดยถูกสร้างขึ้นเพื่อปิดช่องโหว่ความเสี่ยงจากการเข้าถึงเนื้อหาที่ไม่มีการระบุตัวตน (Anonymous Access) หรือการรั่วไหลของเอกสารความลับระดับองค์กรเมื่อส่งไปยังโมเดลภาษาภายนอก (Cloud LLMs)

เส้นทางการไหลของข้อมูลผ่าน UCF Pipeline (Data Flow Path)

1. INPUT
คำขอค้นข้อมูล (Recall) / Task Execution
→
2. CLASSIFY & TAG
สกัด Tags ด้วยลอจิกอัตโนมัติ (Classifier Pack)
→
3. POLICY MATCH
PDP ตรวจสิทธิ์ 4-Tuple ด้วยกฎ YAML (ABAC)
→
4. ENFORCE (PEP)
คัดข้อมูลออก / บังคับใส่ PIN / ลบข้อมูล PII

2. โมเดลข้อมูล 4-Tuple และ Attribute Bag (Request Shape & Attributes)

เพื่อความคล่องตัวและเป็นระบบสากลในการตรวจสอบสิทธิ์ UCF ปลดล็อกสัญกรณ์ตรวจสอบแบบ Hardcoded และหันไปใช้โมเดลโครงสร้างข้อมูลอิงตามมาตรฐาน XACML / Cedar Policy ที่ได้รับการรับรองระดับอุตสาหกรรม:

S - SUBJECT

ผู้สร้างคำขอหรือตัวตนเอเจนต์ พกพาข้อมูลประจำตัว สิทธิ์ผู้ใช้งาน และระดับ Tier ของ Agent (T1/T2/T3)

R - RESOURCE

ข้อมูลหรือวัตถุที่ระบบพยายามจะกระทำ เช่นไฟล์ อะตอม (Atoms) หรือซอร์สโค้ดในพื้นที่จัดเก็บ

A - ACTION

ปฏิบัติการที่กำลังตรวจสอบ เช่น read, modify หรือ expose-to-llm

C - CONTEXT

บริบทของสภาพแวดล้อม เช่นพิกัดเวลาปัจจุบัน หมายเลข Trace ID และข้อมูลสภาพพารามิเตอร์เครือข่าย

Attribute Bag — กลไกสัญกรณ์แบบเปิดกว้าง (Open-Schema Attributes)

เพื่อรองรับความเปลี่ยนแปลงและการทำงานข้ามโปรเจกต์โดยไม่ต้องแก้ไขตัวเอ็นจิ้น ตัวแปรหลักของความปลอดภัยจะถูกเก็บในรูปแบบคีย์-ค่า (Key-Value Bag) แบบอิสระ แต่มีการแนบข้อมูลประวัติแหล่งที่มา (Provenance metadata) เสมอเพื่อป้องกันสัญกรณ์หลอน (Hallucinative Metadata):

  • __source: ระบุนามปากกาของทูลหรือ Classifier ที่เป็นคนกำหนด Tags นั้น
  • __confidence: ระบุค่าคะแนนความแม่นยำ (0.00 – 1.00) ของข้อมูลนั้นเพื่อนำไปสอดประสานกับระดับความปลอดภัย
ตัวอย่างโครงสร้างข้อมูลการตรวจสอบ (Request 4-Tuple Schema) JSON 
{
  "subject": {
    "id": "USER--ICT-98A3",
    "kind": "user",
    "attributes": {
      "tier": "T2",
      "authorized_departments": ["engineering", "security"]
    }
  },
  "resource": {
    "id": "gks/adr/ADR-0043-SECURITY-GATE.md",
    "kind": "atom",
    "attributes": {
      "domain": "security",
      "has_secret": true,
      "confidentiality": "restricted"
    }
  },
  "action": {
    "name": "expose-to-llm"
  },
  "context": {
    "currentTime": "2026-05-25T07:15:00+07:00",
    "traceId": "TR-9823-GKS"
  }
}

3. เครื่องมือตรวจสอบนโยบายความปลอดภัย (ABAC Policy Engine: PDP & PEP)

สถาปัตยกรรม UCF บังคับใช้ความมั่นคงปลอดภัยผ่านระบบแยกแยะส่วนตัดสินใจและบังคับใช้ (Separation of Concerns) ตามมาตรฐานสากล:

pdp Policy Decision Point (จุดประเมินผล)

เป็นฟังก์ชันบริสุทธิ์ (Pure function) ห้ามทำงานเชื่อมโยงกับฐานข้อมูลหรือเครือข่ายภายนอก (No I/O) มีหน้าที่รับอินพุต 4-Tuple และตรวจจับตามกฎที่มีการโหลดจากไฟล์ YAML ในระบบเพื่อส่งผลลัพธ์กลับแบบรวดเร็วระดับ <1ms

  • Hot-reload: โหลดเงื่อนไขนโยบายใหม่โดยไม่มีการปิดการทำงานของระบบ
  • Audit logs: บันทึกรายละเอียดการทำงานและเหตุผลที่บล็อกสิทธิ์ลงระบบตรวจสอบ

pep Policy Enforcement Point (จุดบังคับใช้)

ติดตั้งอยู่ตามจุดขนส่งของระบบ (Transport layers) เช่น Web UI, Express middleware, หรือเครื่องมือ MCP มีหน้าที่ส่งคำขอไปยัง PDP และทำหน้าที่บังคับใช้ตามผลลัพธ์การตัดสินใจ (Deny/Permit)

  • Shadow mode: ประเมินสิทธิ์และส่งบันทึกในแบคกราวด์โดยไม่มีการบล็อกงานจริง เพื่อทำการทดสอบระบบล่วงหน้า
  • Default Posture: สลับโพสเจอร์ความปลอดภัยจาก Shadow เป็น Default-deny ทันทีในส่วนที่มีความอ่อนไหว

นโยบายแบบโครงสร้างข้อมูล (Policy as Data YAML)

ไฟล์นโยบายจัดเก็บแบบแยกชิ้นในหมวดหมู่ policies/*.yaml และควบคุมระดับความถูกต้องตามโมเดลการประเมินลอจิก:

ตัวอย่างไฟล์นโยบาย (policies/60-coding-domain.yaml) YAML 
id: policy/60-coding-domain
name: Coding Domain Security Policy Set
rules:
  - id: restrict-entrypoint-to-t3
    description: "อนุญาตเฉพาะเอเจนต์สัญชาตญาณชั้นนำ (T3) หรือมนุษย์ผู้ควบคุมแก้ไขไฟล์ทางเข้าหลัก"
    effect: deny
    conditions:
      - operator: lessThan
        subject.attributes.tier: T3
      - operator: equals
        resource.attributes.is_entrypoint: true

  - id: permit-test-for-t1
    description: "อนุญาตให้เอเจนต์ตัวเขียนงานย่อย (T1) อ่านและสร้างไฟล์ทดสอบได้"
    effect: permit
    conditions:
      - operator: equals
        subject.attributes.tier: T1
      - operator: equals
        resource.attributes.is_test: true

4. การควบคุมขอบเขตบริบทของซับเอเจนต์ (Subagent Context Scoping & Escalation)

การสปอว์นหรือการแยกงานส่งต่อให้เอเจนต์ย่อย (Subagent) รันสคริปต์เฉพาะ ต้องเป็นไปตามหลัก **POLA (Principle of Least Privilege)** หรือการให้อำนาจขั้นต่ำสุดที่ระบบระบุ เพื่อป้องกันปัญหาการเข้าถึงสิทธิ์เกินความจำเป็นและการทำลายฐานความรู้

STAGE 1
Parent Declarative Scoping

เอเจนต์ตัวนำ (Parent Agent) จะระบุขอบเขตความพร้อมสำหรับ Subagent โดยใส่พารามิเตอร์ excludes หรือ eligible ชัดเจนในการประมวลผล

STAGE 2
Deterministic Filtering

ตัวคุมบริบท (Composer PEP) จะประเมินและคัดแยก Resources หรือความรู้อื่นที่ไม่เข้าเกณฑ์ของ Task Scoping ทิ้งไปก่อนที่จะโหลดเนื้อหาเข้า LLM Prompt

STAGE 3
Escalation Flow

หาก Subagent เกิดความไม่แน่ใจหรือไม่สามารถรันงานต่อได้เนื่องจากขาดสิทธิ์ ระบบมีคำสั่ง escalate() เพื่อยื่นคำร้องขอกว้างสิทธิ์ต่อมนุษย์โดยไม่มีคำสั่งเงียบหาย (No Silent Failure)

เกณฑ์การตัดสินใจแบบมีผลกระทบความคลาดเคลื่อน (Escalation Formula)

ระบบจำกัดความเสี่ยงอย่างเป็นระบบด้วยเงื่อนไขความพร้อมใช้งาน: เมื่อเอเจนต์ย่อยต้องการสิทธิ์เพิ่มเติม (เช่นสิทธิ์เข้าถึงพาร์ทเนอร์ API) PEP จะทำการประเมินความสอดคล้อง หาก PDP ยื่นสิทธิ์ Deny แต่ระดับความจำเป็นของโปรเจกต์อยู่ในขั้นสำคัญ จะส่งเสียงสัญญาณยื่นประวัติขอปลดล็อกสิทธิ์ผ่านรหัสยืนยัน PIN เป็นขั้นถัดไปทันที

5. ความปลอดภัยเชิงสถาปัตยกรรมและการปกป้องข้อมูล PII (Security & PII Redaction)

ข้อมูลส่วนบุคคล (PII) และความลับด้านเทคนิคของระบบรันไทม์ (Credentials/API Keys) จะต้องถูกสแกน คัดแยก และลบคำอ่อนไหวออกไปโดยอัตโนมัติ เพื่อรักษาความเป็นสากลและความคุ้มกันระดับสูงสุดก่อนประมวลผล:

Zero Cloud Exposure

ระบบแสกนเนื้อหาลึก (Security Classifier) จะดักจับพิกัดความลับ เช่น คีย์ AWS, Stripe หรือ OpenAI ด้วยระดับความแม่นยำสูง (Entropy & Pattern checks) หากพบข้อมูลดังกล่าว ระบบจะทำหน้าที่ดักตัด (Circuit Breaker) บล็อกการส่งข้อมูลไปยังเซิร์ฟเวอร์คลาวด์ทุกประเภททันที

Pseudonymization Safety

เพื่อป้องกันการระบุพิกัดตัวตนของผู้สร้างคำขอ ตัวแปรอ่อนไหว เช่น หมายเลขบัตรประชาชน อีเมล หรือชื่อจริงในระบบ Attribute ของ Resource หรือประวัติตัวตนของ Subject จะถูกแปลงเป็นนามแฝง (Pseudonymization) หรือลบออกไปจาก Prompt ก่อนเริ่มทำธุรกรรมประมวลผล

Step-Up Auth (PIN-Gated)

สำหรับปฏิบัติการที่มีความเสี่ยงสูงระดับองค์กร (High-stakes) เช่น การลบ/แก้ไขกฎความจำความคุปลอดภัยถาวร ระบบจะออกข้อบังคับเรียกร้องขอประจุการประเมินยืนยันตัวตนพิเศษ (Step-up Auth) ในระดับผู้ใช้งานจริง (Human-in-the-loop) โดยบังคับให้ใส่รหัสความคุ้มกัน PIN เสมอ

ข้อกำหนดการแสกนและการทำงานด้วย Classifier Plugins

Classifier ID เป้าหมายที่ดักจับ (Detection Targets) ประเภท Attributes ที่ส่งคืน ผลตอบรับด้านนโยบาย (Action)
domain/coding ไฟล์ซอร์สโค้ด Next.js, คีย์เวิร์ด CLI, ไฟล์ทดสอบ language, is_test, is_entrypoint บล็อก T1 ไม่ให้เขียนทับไฟล์ทางเข้าหลัก
domain/security คีย์ AWS, โทเค็นคลาวด์, โครงสร้างรหัสส่วนบุคคล has_secret, encryption_level บล็อกการส่งออกนอกเครื่องโลคอล
domain/task สถานะ ISSUE, ลำดับ RUNBOOK, ข้อมูลอุบัติการณ์ issue_priority, issue_status, assignee บังคับใส่ PIN ในทุกๆ การยุ่งเกี่ยวกับงานระดับวิกฤต
