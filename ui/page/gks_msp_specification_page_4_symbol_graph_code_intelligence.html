Symbol Graph & Code Intelligence

ข้อกำหนดระบบแยกวิเคราะห์โครงสร้างโค้ดเชิงความสัมพันธ์ (Structural Code Representation) ของระบบ GKS และ MSP เพื่อแปลงซอร์สโค้ดดิบ (TypeScript, Python, COBOL) ให้กลายเป็น Knowledge Graph และติดตามผลกระทบของระบบในระดับสากล

1. แนวคิดหลักและขอบเขตการทำงาน (Concept & Scope)

ในขณะที่แกนความรู้ Atomic System จัดการเอกสารและการตัดสินใจระดับสูง (เช่น CONCEPT, ADR, FEAT) ระบบของ Symbol Graph จะถูกออกแบบขึ้นมาเพื่อแสดงโครงสร้าง "ตามความเป็นจริงในซอร์สโค้ด" ปลดล็อกปัญหาเรื่อง Agent ค้นหาโค้ดไม่เจอหรือประเมินขอบเขตความเสียหายในการรีแฟกเตอร์พลาด (Blast Radius Error)

ตรรกะเปรียบเทียบระหว่างการตรวจสอบไวยากรณ์และการวิเคราะห์เชิงโครงสร้างความรู้

มิติการเปรียบเทียบ ระบบเดิม (Flat/ctags Index) ระบบกราฟ GKS/MSP Symbol Graph
รูปแบบการจัดเก็บ (Format) Flat Key-Value (Symbol Name → File) Directed Acyclic Graph (SQLite + JSONL)
ความตระหนักรู้สัญชาตญาณ (Semantic Awareness) ไม่มี สนใจเฉพาะคำและไวยากรณ์เชิงเดี่ยว ตระหนักรู้ประเภทเฟรมเวิร์ก Next.js Pages, Routing และความสัมพันธ์ ORM DB
การจัดกลุ่มการทำงาน (Grouping) อิงตามโครงสร้างโฟลเดอร์แบบดิบ (Physical) จัดกลุ่มชุมชนการทำงาน (Functional Communities) ด้วย Leiden Algorithm
การสแกนหลายภาษา (Polyglot scan) จำกัดเฉพาะภาษาหลักหรือต้องลงทูลเพิ่ม รองรับข้ามเจเนอเรชัน (TS, Python และ Regex-based Legacy COBOL)

การแบ่งแยกสิทธิ์ความรับผิดชอบระว่าง GKS และ MSP

ระบบจำกัดขอบเขตการทำงานอย่างเป็นระบบ: GKS Backend (Rust/napi-rs) มีหน้าที่จัดเก็บและสืบค้นความสัมพันธ์เชิงโครงสร้างแบบไร้ทิศทาง (Graph Primitives) ส่วน MSP Core จะรับผิดชอบการทำดนตรีสัญจรเพื่อแปลโค้ดต้นฉบับออกมาเป็นกราฟ การทำวิจัยพฤติกรรมผ่านท่อประมวลผล 12 ขั้นตอน และการรันเครื่องมือประเมินสิทธิ์ทรานสปอร์ตต่างๆ ในระดับแอปพลิเคชัน

2. ท่อประมวลผลกราฟสัญลักษณ์ 12 ขั้นตอน (The 12-Stage Symbol Graph DAG)

เพื่อสกัดความรู้อย่างเสถียรและไร้จุดผิดพลาด (Deterministic) ตัววิเคราะห์ของ MSP จะทำการสแกนคลังรหัสและแปรสภาพด้วยท่อประมวลผลที่เป็นลำดับขั้นตอนในรูปแบบของ Directed Acyclic Graph (DAG) ทั้งหมด 12 ระยะ:

ระยะสแกนและสกัดขั้นต้น (Static Base Extraction)

  • Stage 1: Scan - สำรวจเส้นทางรหัสใน Repository และวิเคราะห์สถิติไฟล์เบื้องต้น
  • Stage 2: Structure - ขึ้นโครงสร้างแผนภูมิโฟลเดอร์จำลองขอบเขตเชิงกายภาพ
  • Stage 3: Markdown Parse - แปลงข้อมูล wikilinks และ frontmatter จาก Atoms เพื่อนำทางเนื้อหาความจำร่วม

ระยะถอดแยกและจำลองสัญลักษณ์ (Multi-Language AST)

  • Stage 4: COBOL Parse - วิเคราะห์ระบบ Legacy นอกขอบเขตสมัยใหม่ผ่าน Regex Engine ดักจับ PROGRAM-ID และ DIVISION
  • Stage 5: Symbolic Parse - แปลงโค้ด TS/JS และ Python เป็นสัญกรณ์วิทยากรณ์ (AST) ด้วย TypeScript Compiler API และ Tree-sitter

ระยะถอดรหัสเชิงสัญชาตญาณความจำ (Framework Recognition)

  • Stage 6: Routes - ตรวจจับหน้าเชื่อมต่อ API และเส้นทางสัญจร (Next.js App Router endpoints)
  • Stage 7: Tools - สกัดพารามิเตอร์และการนิยามเครื่องมือในระบบ MCP (Model Context Protocol)
  • Stage 8: ORM models - เชื่อมผังกายภาพจาก Prisma Schema หรือตารางฐานข้อมูลระดับ Entity

ระยะลอจิกระดับเชื่อมโยงและจัดกลุ่มชุมชน (Community & Process Tracing)

  • Stage 9: Cross-File Resolution - คลี่ตรรกะความสัมพันธ์นำเข้า-ส่งออกรหัส (Imports/Exports) ข้ามขอบเขตไฟล์
  • Stage 10: MRO (Method Resolution Order) - คลี่ปมสืบทอดสัญชาตญาณคลาสและตัวต้นแบบ
  • Stage 11: Communities - คำนวณคลาสจัดกลุ่มสัญลักษณ์ด้วย Leiden Algorithm อิงตามการทำงานที่ยึดโยงกัน
  • Stage 12: Processes - แกะรอยประจุวิถีการเรียกใช้งาน (Execution Flow) สถิติต่างๆ เพื่อส่งต่อไปเป็นหน่วยการเรียนรู้ให้ MLL

3. ความตระหนักรู้เชิงเฟรมเวิร์ก (Framework-Aware Symbol Indexing)

หัวใจหลักของเครื่องมือวิเคราะห์กราฟสัญลักษณ์ใน GKS/MSP ไม่ใช่แค่การอ่านความสัมพันธ์โครงสร้างไวยากรณ์ทั่วๆ ไป แต่เป็นการเปลี่ยนการอ่านให้สอดรับกับแนวคิดเชิงลึกของแอปพลิเคชัน (Semantic-level Entities):

Next.js Routing Engine

แอนิเมเตอร์ตัววาดกราฟจะเปลี่ยนวิถีไฟล์ page.tsx หรือ route.ts เป็นโหนดพิเศษ Page หรือ Route ทันที โดยเชื่อมต่อความสัมพันธ์ไปหา URL ปลายทางผ่านเส้นเชื่อม RENDERS_AT และผูกสิทธิ์ HTTP methods ทั้งหมดผ่าน HANDLES

ORM Model Semantics

ระบบแอนทิเนสสแกนไฟล์ schema.prisma เพื่อจำลองโหนด Entity ขึ้นมาแทน โดยเชื่อมความสัมพันธ์แบบข้ามขอบเขตตาราง (References) เพื่อให้เอเจนต์เข้าใจแผนผังความเสียหายทั้งหมดเมื่อมีการโยกย้ายหรือแก้ไขคอลัมน์ในโมเดลความจุข้อมูล

MCP Tools Registration

สืบค้นลักษณะการเขียนประกาศลงทะเบียนเครื่องมือภายนอกผ่าน AST ของ TypeScript (เช่นประโยค registerTool({...})) และทำการแมปข้อมูลเพื่อสกัด Schema ของอินพุตออกมาแนบลง Attributes ช่วยให้โมเดลประมวลผลพารามิเตอร์การเรียกทูลได้แม่นยำขึ้นอย่างมีเสถียรภาพ

การค้นหาพิกัดผลกระทบเชิงลึก (Blast Radius Analysis via Cypher v0)

หากนักพัฒนาหรือเอเจนต์ย่อยมีความจำเป็นต้องแก้ไขฟังก์ชันสำคัญในระบบ (เช่นแก้ไขตรรกะการคำนวณคะแนนใน calculateDrift) ระบบสามารถส่งคำขอ Cypher ในระดับ GKS Graph Engine เพื่อตรวจพิกัดความเสี่ยง (Trace Upwards) ได้ทันที:

ตัวอย่างคิวรี Cypher v0 สำหรับวิเคราะห์ขอบเขตความเสียหาย Cypher 
MATCH (target:Symbol { name: "calculateDrift" })
MATCH (affected:Symbol)-[r:CALLS*1..3]->(target)
RETURN affected.name, affected.file, length(r) AS hops
ORDER BY hops ASC

4. การติดตามกระบวนการทำงานและผังกราฟปฏิสัมพันธ์ (Process Tracing & Live Call Graph)

เพื่อให้วิถีการเรียกใช้ฟังก์ชันข้ามไฟล์มีความโปร่งใสและแสดงเป็นมุมมองเสมือนจริงในบราวเซอร์ ตัวจำลองความรู้ของ GKS และ MSP จะทำการพอร์ทผลลัพธ์ผ่านไลบรารี Cytoscape.js ด้วยโครงสร้างภาพกว้างที่เป็นแบบปฏิสัมพันธ์:

ข้อกำหนดพฤติกรรมการควบคุมมุมมอง UI (Visual Controls)

  • Hover Dimming (จางแสง): เมื่อนำเมาส์ไปชี้เหนือโหนดสัญลักษณ์ใดๆ โหนดและเส้นความสัมพันธ์ภายนอกวิถีการเชื่อมโยงจะปรับระดับ Opacity ลงเหลือ 0.1/0.2 ทันทีเพื่อโฟกัสจุดสำคัญ
  • Interactive Pan & Zoom: รองรับการลากกวาดและการย่อขยายแบบสมบูรณ์บนอุปกรณ์พกพา โดยคงขอบเขตขนาดฟอนต์ไว้ในอัตราที่อ่านออก
  • Automatic Cose Layout: จัดเรียงจุดและกลุ่มต่างๆ อัตโนมัติด้วยแรงผลักตามธรรมชาติ โดยหลีกเลี่ยงการวาดเส้นข้ามทับจุดอื่นเมื่อรันระบบครั้งแรก

การสกัดข้อมูลลงแผงรายชื่อ (Details Panel Attributes)

เมื่อทำการคลิกเลือกโหนดสัญลักษณ์ใดๆ แผงควบคุมสถิติตัวแปรทางขวา (#cy-info-panel) จะทำการคลี่ตรรกะข้อมูลเพื่อดึงรายละเอียดเหล่านี้:

Symbol Type function | class | route | entity
K-Impact Index 0.00 – 1.00 (Scalar)
Functional Group Community_ID (Leiden class)
File Reference relative_path:line

5. กฎความถูกต้องเชิงโครงสร้างและตัวติดตาม (Structural & Trace Invariants)

ในการสกัดข้อมูลกราฟ ระบบจะรันการตรวจสอบผ่านอินวาริแอนต์ (Machine-enforced Invariants) ของ MSP Validator เพื่อป้องกันการบิดเบือนของตรรกะและโครงสร้างความรู้:

รายการระเบียบการควบคุมกราฟ (Graph Invariant Enforcements)

ระเบียบ (Invariant ID) ลักษณะกฎข้อบังคับ (The Constraint Rule) เป้าหมายเพื่อป้องกันความเสียหาย
RULE-ACYCLIC-CONSTRAINT ตรวจสอบกราฟการสืบทอดอะตอม (supersedes) และแผนผังการทดสอบ ต้องไม่มีการสลับวงเวียนกลับมาหาตัวเองเป็นลูป (No Cycles) ป้องกันไม่ให้ LLM หลุดเข้าไปในลูปนึกไม่สิ้นสุด (Context stackoverflow) ขณะอ่านความเชื่อมโยง
RULE-RUNTIME-EXCLUSIVITY สัญลักษณ์หนึ่งๆ ห้ามประกาศ Attributes สวนทางกัน เช่น ห้ามมีแท็กเป็นทั้ง runtime: 'client' และ runtime: 'server' ในเวลาเดียวกัน ป้องกันความสับสนของการเรียกใช้ทูลในรันไทม์จริงข้ามเครื่องลอกคอลและเครื่องระยะไกล
RULE-RENDERS-AT-REQUIRED ทุกๆ สัญลักษณ์ที่มีประเภทประเภทเป็น Page ของ Next.js จะต้องมีเส้นเชื่อม RENDERS_AT ชี้ไปยังโหนดความเชื่อมั่นของ URL ปลายทางอย่างน้อย 1 เส้นเสมอ ป้องกันไม่ให้หน้าเว็บค้างลอย (Dangling pages) โดยหาความเชื่อมโยงในการเข้าชมไม่พบ
RULE-INTEGRITY-OF-REFERENCED ห้ามไม่ให้มีเส้นเชื่อม (Edges) ใดๆ ชี้ไปยังโหนดเป้าหมาย (Nodes) ที่ไม่มีตัวตนอยู่จริงในฐานกราฟ SQLite รักษาระดับสมมูลของข้อมูล และป้องกันปัญหา Null Pointer หรือตัวแปลหลอนระหว่างการค้นคืน
