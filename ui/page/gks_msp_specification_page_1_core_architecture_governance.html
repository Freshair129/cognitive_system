Core Architecture & Governance

เอกสารข้อกำหนดระดับสูง (Meta-Architecture) ที่อธิบายสถาปัตยกรรมแกนกลางของระบบ Genesis Knowledge System (GKS) และ Memory & Soul Passport (MSP) รวมถึงกฎเกณฑ์และวัฏจักรการทำงานพื้นฐานที่บังคับใช้ร่วมกัน

1. สถาปัตยกรรมหลัก (Architecture Models)

1.1 MSP Architecture V2 & Agent-Agnostic Design

MSP (Memory & Soul Passport) ทำหน้าที่เป็น Memory OS (Orchestrator) ที่คั่นกลางระหว่าง Agents (เช่น Claude Code, Gemini CLI, หรือ UI Custom) กับฐานความรู้ GKS (Knowledge Base)

  • Agent-Agnostic: MSP ไม่ผูกติดกับ Agent ใดๆ โมเดลความจำ (Session, Episodic, Vector) และตัวตน (Identity) จะติดตัวไปกับ Passport ไม่ว่าผู้ใช้จะเปลี่ยนไปใช้ Agent เครื่องมือใดก็ตาม (CONCEPT--AGENT-AGNOSTIC)
  • Strict Write Boundary: Agents จะไม่สามารถเขียนข้อมูลลง gks/ โดยตรงได้ ต้องผ่านการเสนอ (Candidate) และการตรวจสอบ (Validator) เสมอ

1.2 Two-Brain Architecture

การแยกที่เก็บความจำเพื่อป้องกันการปะปนของบริบทระหว่างโปรเจกต์ (Global vs Workspace)

สมอง (Brain) ตำแหน่งจัดเก็บ (Path) ข้อมูลที่จัดเก็บ วงจรชีวิต (Lifetime)
Global Brain ~/.brain/ ทักษะที่ใช้ข้ามโปรเจกต์, ตัวตน (Identity), และรูปแบบที่เรียนรู้มา ติดกับเครื่องผู้ใช้ (Machine-local)
Project Brain /gks/ Atoms (ADRs, FEATs), Code references, บริบทเฉพาะโปรเจกต์ ผูกกับ Repository (Commit ลง Git)

2. ระบบความรู้และการจัดเก็บ (Knowledge & Storage Layers)

3-Tier Knowledge Model

การแบ่งระดับความสำคัญของความรู้ (Epistemic Provenance)

  • 1. Safety: กฎห้ามละเมิดเด็ดขาด (Invariants, Guardrails, Proto)
  • 2. Master: สัญชาตญาณแกนกลาง โหลดเข้า System Prompt (เช่น กฎ Doc-to-code)
  • 3. Genesis: ความรู้ประกอบ (Composite) ที่ประกอบจากหลายมิติ

4-Layer Storage Model

โครงสร้างการค้นคืนข้อมูลเมื่อ Agent สั่ง recall()

  • Atomic: นิยามที่ถูกต้อง 100% (ID Match) <1ms
  • Vector: ค้นหาความหมายที่ใกล้เคียง (Semantic) 10ms
  • Episodic: สรุปเหตุการณ์สำคัญในอดีต 10ms
  • Obsidian: ความเชื่อมโยงแบบกราฟและ Fulltext 100ms+

3. ธรรมาภิบาลและวัฏจักรการทำงาน (Governance & Workflow)

3.1 7-Phase Doc-to-Code Flow

กฎ "Doc Before Code" ไม่มีข้อยกเว้น (ยกเว้น Hotfix) การพัฒนาฟีเจอร์ใดๆ ต้องผ่าน 7 ลำดับนี้:

P0

FRAMEWORK / FRAME

กำหนดสถาปัตยกรรมระบบหรือโครงสร้างกฎเกณฑ์ระดับสูง

P1

CONCEPT

กำหนดวิสัยทัศน์ (PRD), ปัญหา และทิศทางกลยุทธ์กว้างๆ

P2

ADR / FEAT / PROTO

บันทึกการตัดสินใจ, สเปกของฟีเจอร์, หรือข้อบังคับ (Invariants)

P3

BLUEPRINT

พิมพ์เขียวแผนการ Implement จริง (แบ่ง Tasks ย่อยลงไฟล์ไหน)

P4

TASK (Microtask)

ไฟล์ T*.task.yaml แตกงานสำหรับส่งให้ SLM / Agent เขียนโค้ด

P5

CODE

แก้ไขลง Source Code จริง (เขียนโค้ด)

P6

AUDIT

บันทึกผลการ Acceptance Test และสิ่งที่ส่งมอบจริง

3.2 Scaling Levels (L1-L3)

ไม่ใช่ทุกงานต้องทำ 7-Phase ระบบบังคับเฉพาะ Artifact ที่จำเป็นตามสเกลงาน

  • L1 (Quick Task): แก้บั๊กเล็ก, พิมพ์ผิด
    บังคับ: ไม่ต้องสร้าง Atom ใช้แค่รอบ Audit สั้นๆ
  • L2 (Feature): ฟีเจอร์ปกติ
    บังคับ: CONCEPT + ADR + FEAT
  • L3 (Major): ระบบใหญ่, กระทบวงกว้าง
    บังคับ: Full chain รวม BLUEPRINT + AUDIT

3.3 Authority Matrix

ใคร (Tier ไหน) มีสิทธิ์เขียนอะไร

  • T1 (Microtasks): เขียนโค้ดตาม Task YAML
  • T2 (Implementer): แตก Task, วางโครงสร้างโค้ด ห้ามเขียน ADR
  • T3 (Architect): เขียน ADR, Blueprint ควบคุมภาพรวม
  • Boss (Human): อนุมัติการเข้าสู่ `gks/`, จัดการ Master Policy

4. อนุกรมวิธานและบล็อกกำเนิด (Taxonomy & Genesis Block)

4.1 Atom Registry as SSOT

ระบบใช้ไฟล์ atom_registry.yaml เป็น Single Source of Truth (SSOT) ในการนิยามหมวดหมู่ (Taxonomy V2.3) เพื่อใช้สร้าง JSON Schema และตรวจสอบ Validation อัตโนมัติ แทนที่จะใช้ LLM อ่านคู่มือ Markdown ยาวๆ ซึ่งช่วยประหยัด Token ได้มหาศาล

4.2 Genesis Block Manifest

GENESIS-- คือหน่วยประกอบ (Composite Unit) ที่รวมเอา Atoms หลายๆ ส่วนเข้าด้วยกันเพื่อให้ Agent เข้าใจความรู้ชุดนั้นแบบสมบูรณ์ โดย 1 Genesis Block (ระดับ Master) ต้องประกอบด้วย 4 ใน 5 มิติหลัก (4D Completeness Check):

ALGO--
Logic/Steps
CONCEPT--
Semantics
FRAME--
Mental Model
PROTO--
Implementation
PARAMS--
(Optional)

Contradiction Policy & Master Promotion

การอัปเกรดความรู้เป็นระดับ Master ห้ามเกิดจากการเขียนอัตโนมัติ (Never Auto-write) ต้องผ่านการอนุมัติแบบมี Human-in-the-loop เท่านั้น และหากเนื้อหาขัดแย้งกับข้อมูลเดิม ระบบบังคับให้ทำ Reciprocal Supersession แบบชัดเจนภายใน Pull Request เดียวกัน (MASTER--ATOM-CONTRADICTION-POLICY).
