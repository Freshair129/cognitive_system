Validation, Automation & Quality Assurance

ข้อกำหนดสำหรับ "ด่านตรวจคุณภาพความมั่นคงสากล" (Universal Quality Gates) ทั้งในระดับเครื่องโลคอลและระบบ CI/CD เพื่อรักษาระเบียบโครงสร้างความรู้ GKS ไม่ให้ปนเปื้อนข้อมูลดิบ ข้อมูลเสื่อมสภาพ หรือพฤติกรรมหลอนจากโมเดลปัญญาประดิษฐ์

1. MSP Validator และกลุ่มกฎต้านการหลอน (Anti-Hallucination Guards)

การสแกนและตรวจสอบโครงสร้างความรู้ผ่านคำสั่ง msp:validate จะทำงานแบบ Stateless & Deterministic (ไม่ใช่โมเดลปัญญาประดิษฐ์ประเมินผล เพื่อป้องกันสัญการณ์วิเคราะห์ที่คลาดเคลื่อน) โดยดักจับและสกัดกั้นทันที (Exit-1) เมื่อพบการทำงานผิดระเบียบดังต่อไปนี้ (ADR--ANTI-HALLUCINATION-RULES):

6 กฎเหล็กพฤติกรรมความคลาดเคลื่อน (6 Core Anti-Hallucination Guards)

รหัสและแท็ก (Rule ID) ลักษณะกฎข้อบังคับ (The Constraint) เป้าหมายเพื่อกำจัดความหลอน (Hallucination Defended)
[dangling-wikilink] ตรวจสอบทุกๆ ลิงก์ภายในเนื้อหา [[ATOM-ID]] และคลังความสัมพันธ์ crosslinks.* ทุกตัว ต้องปรากฏชื่อจริงอยู่ในดัชนี GKS เท่านั้น ป้องกัน "Ghost references" หรือการสร้างโหนดความรู้หลอนที่อ้างอิงถึงเอกสารที่ไม่ได้ถูกสร้างขึ้นจริง
[adr-monotonic] เมื่อสร้าง ADR ใหม่ (เช่น ADR-005) ระบบจะตรวจสอบดัชนี ถ้าพบเลขสูงกว่า (เช่น ADR-007) อยู่แล้วจะส่งสัญญาณบล็อก ป้องกันไม่ให้ LLM แย่งเลข ID ชนกันเอง หรือสุ่มกระโดดเลขเว้นว่างอย่างไร้ระเบียบในคลังความรู้
[no-future-dates] ค่าตัวแปร created_at และ promoted_at ต้องไม่เป็นเวลาที่เกินหน้าไปกว่าเวลาปัจจุบันของรันไทม์ กำจัดอาการ "Timestamp hallucination" ของเอเจนต์ที่ชอบลงวันที่อนาคตเกินความเป็นจริง
[summary-min] สแกนช่อง summary: ของ Frontmatter ต้องมีความยาว 10–300 ตัวอักษร และต้องไม่มีคีย์เวิร์ดหลอกตา เช่น TODO, TBD, FIXME, lorem ipsum ป้องกันความรู้ขยะ (Content Rot) ที่โมเดลทิ้งคำอธิบายสั้นๆ ไร้ความหมายเพื่อประหยัดการทำงานส่งมอบงานค้าง
[id-filename-match] ฟิลด์ id: ใน YAML Frontmatter ด้านในสุดของไฟล์ จะต้องเป็นคีย์สะกดตรงกันกับชื่อไฟล์ (Filename Base) 100% ป้องกันความสับสนเชิงโครงสร้างกรณีที่มีการก๊อปปี้เทมเพลตมาแก้แล้วลืมอัปเดตรหัสแกนกลางด้านใน
[evidence-for-decisions] สำหรับไฟล์ประเภท type: adr ต้องสแกนพาดหัว H2 ด้านใน ต้องครบถ้วนตามสัญญา: Context, Decision, Consequences ป้องกันการตัดสินใจหลอนแบบลอยๆ ไร้การบันทึกประจักษ์พยานเหตุผลสนับสนุนเชิงเทคนิค (Evidenceless Decisions)

2. การตรวจสอบการบิดเบือนของประเภทและการเสื่อมสภาพ (Registry Drift Check & Atrophy Guards)

drift check 2.1 Registry Drift Check (การบิดเบือนหมวดหมู่สากล)

เพื่อรักษาเกียรติยศสูงสุดของการจัดประเภทความรู้ คอนแทรค PROTO--REGISTRY-DRIFT-CHECK บังคับว่า ทุกๆ อะตอมไฟล์ที่นำส่งเข้าสู่ GKS ต้องไม่ทำการเขียนประเภทสะกดเอง (Unregistered Types) โดยดักสแกนฟิลด์ type และ phase เทียบกับพารามิเตอร์ที่นิยามใน Single Source of Truth อย่าง atom_registry.yaml เสมอ:

เกณฑ์การตอบรับแบบเด็ดขาด (Hard Rejection Trigger):
  • มีประเภทที่ไม่ปรากฏในรีจิสทรี เช่น type: custom-draft-prefix
  • มีการระบุ Phase ขัดแย้งกับประเภทหลัก เช่น ไฟล์ประเภท type: adr แต่จงใจระบุเป็น phase: 1 (ความจริงคือกฎระบุเป็น Phase 2 เสมอ)

atrophy check 2.2 Decision Atrophy Guards (การตรวจจับความล้าสมัยของตรรกะ)

เอกสารเชิงกลยุทธ์ (เช่น Roadmap แผนงาน หรือผลลัพธ์ Benchmark) มีสิทธิ์ล้าสมัยและกลายเป็น "ความเชื่อแบบเท็จหลอกตา" (Enforced Lies) หากไม่มีการเข้ามารีวิวซ้ำ ตัวตรวจจับ FEAT--DECISION-ATROPHY-GUARDS จะทำการแสกนหาฟิลด์ valid_until ใน Frontmatter และประเมินดังนี้:

🟡 Warning: Near Expiry (เตือนใกล้หมดอายุ)

หากตัวแปร valid_until มีระยะเวลาเหลือ น้อยกว่า 14 วัน เทียบกับเวลาปัจจุบัน ระบบจะขึ้นประจุสัญญาณแจ้งเตือนแต่ยอมให้ประมวลผลผ่าน

🔴 Violation: Expired (หมดอายุแล้ว)

หากพิกัดเวลาปัจจุบันผ่านเลยหลัก valid_until ไปแล้ว ระบบจะถือเป็นข้อผิดพลาด (Warning ที่อาจยกขึ้นเป็น Error ได้) บังคับให้สถาปนิก T3 มารีวิวและเพิ่มระยะวัน

3. ตัวควบคุมการส่งรหัสอัตโนมัติ (Automation Hooks)

เพื่อให้ธรรมาภิบาลทำงานได้ตั้งแต่ระดับเครื่องนักพัฒนา (Local Shift-Left) ระบบ GKS/MSP ทำการติดตั้งกลไกความปลอดภัยเป็น Git Hooks เฝ้าประตูหลัก 2 จุด:

Pre-commit Hook (ตรวจจับก่อนยืนยันสิทธิ์)

สคริปต์ความปลอดภัยระดับเครื่องลูกข่าย pre-commit-validator.sh จะเฝ้าติดตามคำสั่งคอมมิตของ Git โดยจะแอบดึงเฉพาะไฟล์เอกสารอะตอมที่ถูกจัดคิวไว้ (Staged Markdown Files via git diff --cached) ไปวิ่งผ่านด่านตรวจ MSP Validator

จุดเด่น: แสกนเสร็จสิ้นใน <100ms หากพบเพียงข้อเดียวที่พัง จะดีดตัวกลับไม่ยอมให้จบการคอมมิต

Pre-push Hook (ตรวจจับสายใยก่อนนำส่ง)

สคริปต์เฝ้าประตูระดับเครือข่าย pre-push-verify.sh จะดักจับคำสั่ง Git Push เพื่อรันคำสั่ง gks verify-flow เช็คความต่อเนื่องของสายใยฟีเจอร์ที่ได้รับผลกระทบ (Touched FEATs Chain) ทั้งหมด

จุดเด่น: ยืนยันว่าไม่มีการนำส่งรหัสต้นฉบับ P5 CODE ออกไปโดยปราศจากบลูพริ้นท์ P3 คุมด้านหลัง

4. กลไก Hotfix (Emergency Bypass Protocol)

ในสถานการณ์วิกฤตระบบรันไทม์เกิดข้อผิดพลาดรุนแรง การรอให้เขียนเอกสาร 6 เฟส (P0-P3) อาจทำให้สัดส่วนความเสียหายทางธุรกิจขยายวงกว้าง GKS/MSP จึงเปิดช่องทางพิเศษด้วยกลไก Hotfix Protocol ภายใต้เงื่อนไขควบคุมที่ปลอดภัยและเคร่งครัด:

เส้นชีวิตธุรกรรมด่วนพิเศษ (Hotfix Lifecycle)

1 Open Hotfix (ขอยกรันเวย์สิทธิ์)

ผู้ควบคุมรันคำสั่ง npm run msp:hotfix:open เพื่อสร้างรหัสคิวบั๊กพิเศษและปลดเบรก pre-commit ชั่วคราว

2 Deploy Emergency Code (เขียนโค้ดและส่งขึ้นสายผลิต)

ข้ามขั้นตอนเอกสาร (CONCEPT, ADR, BLUEPRINT) ไปแก้ไขซอร์สโค้ดจริงในระดับ P5 ได้ทันที และยินยอมให้ทำคอมมิตเพื่อ Push ขึ้นเซิร์ฟเวอร์

3 48-Hour Backfill Constraint (ด่านเช็คหนี้เอกสาร 48 ชั่วโมง)

ผู้ควบคุมทีมต้องเขียนเอกสารสบย้อนหลัง (Backfill Atoms) ลงในระบบ GKS ให้ครบถ้วน และเพิ่มลิงก์ความสัมพันธ์ crosslinks.resolves: [HOTFIX--] เพื่อเคลียร์หนี้ความรู้

4 Auto-Lock Gate (ระบบปิดทางด่วนอัตโนมัติ)

หากเวลาพ้น 48 ชั่วโมงแต่ไม่มีเอกสารย้อนหลัง ตัวแรปเกตเวย์จะดักบล็อกไม่ยอมให้ push หรือคอมมิตงานใดๆ ข้ามระบบต่อไปเด็ดขาด

5. กฎสัญญาเครื่องจักร (Machine Invariants: PROTO Rules)

กลุ่มกฎอินวาริแอนต์ (PROTO Predicates) ที่ตัวประมวลผลเช็คอัตโนมัติที่ระดับลึก เพื่อปกป้องระเบียบของ prompt-injection และพฤติกรรมของรันไทม์:

PROTO--MASTER-TOKEN-CAP

กฎคุมโทเค็น System Prompt

เพื่อป้องกันไม่ให้ Master atoms มีเนื้อหายาวเกินไปจนกลืนงบประมาณและขโมยบริบทของเอเจนต์ บังคับนับคำด้วยสัดส่วน BPE Heuristic (คำคั่นด้วยเว้นว่าง × 1.3)

≤ 400 tokens: PASS
> 400 tokens: WARNING
> 600 tokens: REJECT (Error)
PROTO--PHASE-GATES

กฎการจัดเรียง Phase (P0-P6)

สำหรับทุกซอร์สโค้ดไฟล์ใน linked_symbols ของอะตอมระดับ P5 (CODE) ต้องปรากฏชื่อไฟล์นั้นพ่วงอยู่ในเอกสาร BLUEPRINT ระดับ P3 เสมอ

ข้อยกเว้น (Escape Hatch):
ใส่แท็ก skip_blueprint: true ใน Frontmatter เฉพาะเคสโค้ดตัวอย่างภายนอก
PROTO--ALGO-PARAM-COUPLING

กฎสายใยตัวแปรและอัลกอริทึม

เพื่อป้องกันไม่ให้พารามิเตอร์รันไทม์ลอยละลิ่วอย่างไร้ทิศทางและควบคุมความเสถียรของซอร์สโค้ด บังคับตรวจสอบความเชื่อมโยงแบบสองทิศทาง (Reciprocal link)

ALGO--X.tunable_by
ต้องชี้ไปพิกัดหา
PARAM--Y.tunes เสมอแบบครบคู่
