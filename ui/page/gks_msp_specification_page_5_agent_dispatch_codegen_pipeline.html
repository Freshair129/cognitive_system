Agent Dispatch & Codegen Pipeline

สเปกและข้อกำหนดทางสถาปัตยกรรมของตัวรันรหัสไมโครทาสก์ (Microtask Codegen Runner) และการบริหารจัดการคิวการสั่งการ (Agent Dispatch) ภายใต้เกณฑ์ควบคุมทางการเงินและระดับการตัดสินใจของเอเจนต์ (T1/T2/T3)

1. การจัดเส้นทางและการยกระดับเอเจนต์ (Agent Tier Routing & Escalation Flow)

เพื่อจำกัดงบประมาณและประหยัดการใช้โทเค็น (Token Optimization) ระบบหลีกเลี่ยงการใช้โมเดลราคาสูงทำธุรกรรมทั่วไป โดยแยกแยะระดับงานออกเป็น 3 ระดับและใช้อัลกอริทึมเลือกโมเดลตามลำดับชั้นอย่างเคร่งครัด (CONCEPT--AGENT-TIER-ROUTING):

Tier 1 - Executor

Local Codegen / Utils

เหมาะสำหรับงานที่เน้นจำนวนทาสก์เชิงเดี่ยวระดับอะตอม (1 concern per task) และไม่ประเมินสิทธิ์ภาพรวม

Model: Qwen 2.5 Coder 7B
Cost: $0.00 (Local-First)
Access: Local workspace files
Tier 2 - Implementer

Feature / Route Impl

งานเขียนโค้ดฟีเจอร์ระดับกลางที่มีโครงสร้างไฟล์และตรรกะเชื่อมโยงภายนอก หรือต้องการรัน API

Model: Gemini 1.5 Flash / Pro
Cost: Low to Moderate (Cloud)
Access: Project-wide Atoms
Tier 3 - Architect

Core Architecture / ADR

การออกแบบโครงสร้างหลัก การวาง Blueprint การตัดสินใจสำคัญ และการแก้ความขัดแย้งของระบบ

Model: Claude 3.5 Sonnet / Opus
Cost: High (Cloud API)
Access: Full 3-Tier + Global

เส้นทางและขีดจำกัดการยกระดับเมื่อการประมวลผลล้มเหลว (The Escalation Loop)

ระบบใช้อัลกอริทึมตรวจสอบผลลัพธ์ผ่านชุดรันเทส (Acceptance tests) หากทาสก์ของ Tier 1 ไม่ผ่านตามข้อกำหนด ระบบจะไม่ยอมจำนนแบบเงียบหาย (No Silent Failure) แต่จะยกระดับขึ้นไปประมวลผลบนชั้นถัดไปตามลำดับขั้น:

T1 Failure (Max 3 retries)
→
Escalate to T2 (Gemini)
→
If severity ≥ regular: Escalate to T3 (Claude)

2. ระบบติดตามต้นทุนและคลังสถิติการใช้ Token (Cost Tracking & Usage Rollups)

หลังจากการประมวลผลของ dispatch(task) แต่ละครั้ง ระบบจะคำนวณและบันทึกประวัติค่าใช้จ่ายสถิติเพื่อเป็นคลังข้อมูลในการประเมินประสิทธิภาพเชิงตัวเลขผ่านโครงสร้างของ USAGE--DAILY Atoms:

สูตรคำนวณราคาเฉลี่ยต่อทาสก์ (Mathematical Pricing Estimation)

เพื่อให้ได้การประเมินราคาล่วงหน้า (Heuristic pricing) ที่ใกล้เคียงความจริงโดยไม่ผูกมัดกับใบเรียกเก็บเงินปลายงวด:

Cost_USD = ((input_tokens * per_million_input_usd) + (output_tokens * per_million_output_usd)) / 1,000,000

*(หมายเหตุ: สำหรับ Tier 1 ที่เป็นการประมวลผลบนเครื่องโลคอลจะมีค่าใช้จ่ายเป็น 0 เสมอ)*

3. สัญญาไมโครทาสก์คลังโค้ด (The Codegen Microtask Contract)

ไฟล์ทาสก์จำลอง T*.task.yaml จะถูกเขียนขึ้นในช่วง Phase 4 เพื่อสั่งการเอเจนต์ย่อยสำหรับการแปลงผลให้กลายเป็น Source Code ในรันไทม์จริง โดยคิวงานทั้งหมดต้องประกอบด้วยขอบเขตเงื่อนไขดังนี้:

ตัวอย่างสัญญาคิวงานย่อย (gks/microtasks/T1001_build_auth.task.yaml) YAML 
id: T1001_build_auth
parent_blueprint: BLUEPRINT--IDENTITY-LAYER
scale_level: L2
prompt: "เขียนฟังก์ชันสำหรับสร้าง Token และแรปสัญชาติ FFI จาก /gks/native/ โดยห้ามแก้ไขไลบรารีดั้งเดิม"
acceptance:
  test_command: "npm test --workspace=packages/msp -- test/identity/jwt.test.ts"
  required_exports:
    - "generateToken"
    - "verifyToken"
geography:
  - file: "packages/msp/src/identity/jwt.ts"
    role: "implementation"
  - file: "packages/msp/test/identity/jwt.test.ts"
    role: "test"

กระบวนการประมวลผลย่อยหลังรับรหัสจาก SLM (The Strip Pipeline)

เพื่อให้ได้ความแน่นอนในการเขียนทับและป้องกันอาการ "พร่ำเพร้อ" (Extra prose) ของ LLM ผลลัพธ์ที่ได้มาจากโมเดลภาษาจะต้องผ่านท่อสตรีมข้อมูลเพื่อกรองและตัดส่วนประกอบภายนอกตามสัญญาทางสถาปัตยกรรม:

1. Strip Markdown Fences ลบขอบเขตบล็อกโค้ด ```typescript หรือ ``` ออกไปทั้งหมดเพื่อสกัดเอาเฉพาะส่วนที่เป็นรหัสต้นฉบับเพียวๆ
2. Cut Prefix Commentary ค้นหาบรรทัดนำเข้าแรกที่มีคำสั่งหลัก (เช่นคำสั่ง import หรือประกาศ const) และดักตัดข้อความเกริ่นนำอธิบายก่อนหน้าทิ้งทั้งหมด
3. Cut Suffix Commentary ค้นหารหัสปิดปีกกาแถวสุดท้ายที่มีความเชื่อมโยง (}) และทำการดักตัดข้อความทิ้งท้ายหรือความเห็นอภิปรายด้านหลังออกไปทั้งหมด

4. กฎสกัดคำและคำต้องห้ามในการเขียนรหัส (Forbidden Patterns Check)

เพื่อป้องกันปัญหาการทิ้งคอมมิตเปล่า (TODO punts) หรือโครงสร้างลักไก่ข้ามข้อบังคับด้านความปลอดภัย ทุกๆ รหัสผ่านการเจเนอเรตจะรันสแกนผ่านตัวแปร Pattern-matching หากตรวจพบคำสั่งตามเงื่อนไขเหล่านี้ระบบจะบล็อก (Hard Rejection) ไม่ให้คอมมิตเด็ดขาด:

เป้าหมายที่ดักจับ (Regex Pattern) ประเภทความปลอดภัย (Severity) เหตุผลทางวิศวกรรม (Engineering Rationale)
/export\s+default/ hard reject ระบบใช้สัญกรณ์ Named Exports เคร่งครัดในการทำ FFI และประเมินสิทธิ์ความน่าเชื่อถือข้ามไฟล์ ห้ามสร้าง Default Export เด็ดขาด
/req\.body/ hard reject ห้ามเข้าถึง Body ดิบของ Request โดยตรง ต้องผ่านตัวกรองสิทธิ์และ Attribute Bag เสมอเพื่อตรวจสอบ ABAC และ PII
/\/\/.*(TODO|FIXME|XXX)/i hard reject เอเจนต์ย่อยห้ามปัดภาระงานค้างไว้เป็น Todo ในโค้จรันไทม์จริง หากทำไม่ได้ระบบบังคับให้รันคำสั่งยื่นคำร้องยกรันเวย์สิทธิ์แทน
/console\.log/ soft warning อนุญาตเฉพาะส่วนสลักตรวจประวัติการรันไทม์ (Trace) นอกเหนือจากนั้นระบบแนะนำให้เปลี่ยนไปใช้ Custom Observability API ของ GKS
/process\.env/ soft warning เลี่ยงการเข้าถึงสภาพแวดล้อมโดยตรงในระดับ Routing แนะนำให้ดึงข้อมูลผ่านการใช้ระบบ Configuration Loader ของ MSP ที่ปลอดภัยกว่า

5. การทดสอบแซนด์บ็อกซ์และการเชื่อมต่อโลคอลโมเดล (Vitest Sandbox & Ollama Integration)

ตัวประมวลผลไมโครทาสก์ของ GKS และ MSP ทำงานภายใต้สถิติ **Local-First** เคร่งครัด เพื่อให้มั่นใจได้ว่าระบบสามารถทดสอบรหัสต้นฉบับของเอเจนต์ได้ทุกเวลาแบบไม่พึ่งพาเน็ตเวิร์กคลาวด์ภายนอก:

Vitest Sandbox Isolation

เมื่อ SLM พ่นโค้ดเวอร์ชันดราฟต์ ตัววิเคราะห์จะแอบก๊อปปี้ไปวางในพิกัดพื้นที่แซนด์บ็อกซ์ที่แยกจากตัวซอร์สโค้ดจริงชั่วคราว ก่อนสั่งรันคำสั่ง vitest run แบบปิดการมองเห็นและจำกัด I/O เพื่อตรวจสอบว่าผ่านเทสเคสทั้งหมดในระดับความมั่นใจ 100% จึงจะยอมให้รวบพาร์คความจำ

  • Auto-cleanup: ลบไฟล์แซนด์บ็อกซ์ทิ้งทันทีเมื่อเทสเสร็จสิ้น
  • Concurrency Protection: รันเทสด้วยรหัสสุ่ม UUID เพื่อแยกแยะประเด็น

Ollama Local SLM Client

ตัวคุมระบบ (createSlmClient) สามารถต่อเข้ากับพอร์ต HTTP โลคอลของโปรแกรม **Ollama** สำหรับการเรียกตรรกะแบบไม่ต้องอาศัยกุญแจ API ภายนอก และลดต้นทุนการใช้คลาวด์ได้อย่างมหาศาล สอดคล้องกับพฤติกรรม Local-First:

Env Host: OLLAMA_HOST (default http://127.0.0.1:11434)
Env Model: OLLAMA_MODEL (default qwen2.5-coder:7b)
