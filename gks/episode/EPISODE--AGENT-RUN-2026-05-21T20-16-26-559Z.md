The **Identity Engine** in the Genesis Knowledge System (GKS) serves as the "soul" of the agent, providing a stable, persistent persona across sessions. It is designed around the concept of a **Passport** that externalizes the agent's identity, allowing it to move across heterogeneous environments while maintaining consistency in name, tone, and behavior.

### 1. Conceptual Framework
The identity system is built on two foundational cognitive pillars:
*   **Identity Layer (`CONCEPT--IDENTITY-LAYER`)**: Defines the three core dimensions of an agent's identity:
    *   **Profile**: Stable identifying facts (`name`, `role`, `tier`, `originStory`).
    *   **Voice**: Communication style (`tone`, `formality`, `languagePreference`, `responseCadence`).
    *   **Preferences**: Scoped runtime overrides (`verbosity`, `code_block_style`) with optional TTL.
*   **Ego Death Protocol (`COGNITIVE--EGO-DEATH-PASSPORT`)**: A behavioral mandate requiring the agent to "surrender" any internal model-specific persona in favor of the identity injected from the passport. The agent must acknowledge itself as a "blank slate" and treat the passport as the highest authority for its self-reference and behavior.

### 2. Technical Implementation
The system is implemented within the `@freshair129/msp` package (`packages/msp/src/identity`) and follows a layered resolution strategy:

*   **Layered Resolution (`ALGO--IDENTITY-RESOLUTION`)**:
    *   **Global**: Base identity stored in `~/.msp/identity.json`.
    *   **Project**: Sparse overrides stored in `.brain/msp/projects/<namespace>/identity.override.json`.
    *   **Merge Logic**: A shallow merge is performed at the top-level keys (`profile`, `voice`, `preferences`), where project-specific tweaks override global defaults.
*   **Storage & Atomicity**: Identity is stored as a versioned JSON file (Schema v1). Writes are atomic, using a "temp-then-rename" pattern to prevent data corruption.
*   **Programmatic API (`PROTOCOL--IDENTITY-API`)**:
    *   `getIdentity(ns)`: Loads and merges the identity for a given namespace.
    *   `setProfile()`, `setVoice()`, `setPreference()`: Atomically updates specific sectors of the identity.

### 3. Key Findings
*   **Separation of Concerns**: Identity is treated as **execution state**, not durable knowledge. Therefore, it is stored in the `.brain/` or user home directory rather than the `gks/` atom store.
*   **UCF Integration**: Recent updates (Phase 4 of the Universal Context Framework) have expanded the identity profile to include security-related fields such as `roles`, `clearance` level, and `mfaStatus`.
*   **Migration Path**: The system includes a legacy migration layer that transitions old project-local identities into the new global/project split.

In summary, the Identity Engine ensures that an agent like Gemini CLI can "know who it is" (e.g., "EVA", a T3 research assistant) and maintain a consistent "voice" (e.g., "analytical and warm") regardless of the underlying LLM provider or session state.
